InMoment camera screen backend rules

Этот файл фиксирует правила интеграции Flutter camera screen с backend.

Основная архитектурная идея

Главный экран приложения реализован как camera hub, а не обычная социальная лента.
Поэтому backend flow для камеры должен быть коротким, явным и без лишних ветвлений.

Для публикации медиафайла использовать только такой порядок

1. Получить active group
2. Получить presigned upload URL
3. Загрузить файл напрямую в storage
4. Вызвать publish endpoint
5. Обновить group feed

Чего не делать во Flutter

1. Не генерировать StorageKey на клиенте
2. Не вызывать publish без presign
3. Не пытаться угадать публичный URL медиафайла на клиенте
4. Не строить photo card вручную после upload
5. Не использовать простой feed endpoint как основной экран

Что считать успешной публикацией

Публикация считается успешной только если:
1. файл успешно загружен в storage
2. publish endpoint вернул успешный ответ
3. в ответе получен PhotoId
Backend возвращает PhotoId. Текущий Flutter flow может не использовать его напрямую и после успешного publish обновляет feed/paged.

Что делать после успешной публикации

Минимальный правильный путь:
1. перезагрузить GET /api/groups/{groupId}/feed/paged

Допустимый путь для навигации:
1. получить PhotoId из publish response
2. открыть GET /api/photos/{photoId}

Что backend уже гарантирует

1. Проверку членства пользователя в группе
2. Проверку допустимого content type
3. Проверку размера файла
4. Проверку соответствия StorageKey текущему user/group
5. Создание Photo в базе
6. Realtime уведомление о смене feed

Что должен гарантировать Flutter

1. Передавать корректный GroupId
2. Передавать ContentType, совпадающий с реальным файлом
3. Передавать SizeBytes после реального выбора файла
4. Не кешировать presigned upload URL слишком долго
5. Обрабатывать ошибки presign и publish отдельно
